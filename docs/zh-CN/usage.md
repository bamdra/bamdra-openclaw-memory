# bamdra-memory 使用指南

## 别把它当成“数据库功能”，把它当成“连续性增强”

`bamdra-memory` 的价值不在于多了几个工具名，而在于对话会变得更像和一个持续工作的搭档协作。

最理想的体验是：

- 你中途被别的话题打断，也不用担心前面的思路丢掉
- 你说“记住这个”，后面真的能用上
- 你重新聊回之前的话题时，agent 能自然接上
- 这些恢复都发生在当前 agent / 当前用户会话边界内

## 一个典型使用过程

假设你正在和 OpenClaw 一起规划生活和工作：

### 场景一：隐式切换又自然接回

你说：

> 下个月想在国内找个地方短途旅游。

接着又说：

> 如果去成都，先吃什么比较值？

这时突然出现一个完全不同的话题：

> 我刚收到一个工作邮件，帮我写个礼貌回复，说我明天上午发文件过去。

等邮件处理完，你继续问：

> 继续说旅游。如果只有一个周末，成都和杭州哪个更合适？

理想效果：

- 旅游主线还能自然接上
- “吃什么”这条支线仍然和旅游主线保持关联
- 工作邮件不会污染后面的旅游建议
- 旅游和工作之间既能隐式切换，也能在需要时显式切换

### 场景二：显式要求记住一个事实

你说：

> 请记住我坐火车更喜欢靠窗的位置。

理想效果：

- 这个偏好会被保存下来
- 后面聊旅游、订票时不需要你再重复一次

## 实际上有哪些工具

当前工具集就是下面这 5 个：

- `memory_list_topics`
- `memory_switch_topic`
- `memory_save_fact`
- `memory_compact_topic`
- `memory_search`

### 这 5 个工具各自做什么

它们的定位可以理解成：

- `list_topics`：看现在有哪些话题分支
- `switch_topic`：把当前上下文切到某个已存在的话题
- `save_fact`：把一个稳定事实写进长期记忆
- `compact_topic`：强制刷新某个话题的摘要
- `search`：从已存的话题和事实里找答案

其中：

- `list_topics` 和 `search` 主要是“读”
- `switch_topic`、`save_fact`、`compact_topic` 会改变状态或写入 SQLite

### `memory_list_topics`

作用：

- 列出当前 `session` 下已经存在的 topic 分支
- 返回每个 topic 的标题、摘要、标签等信息
- 告诉你哪个 topic 当前处于激活状态

典型触发时机：

适合什么时候用：

- 想看当前 session 已经有哪些上下文线程
- 不确定之前那段对话现在被归到了哪里
- 用户说“刚才那个话题”“之前那条线”“我们现在有几个分支”
- agent 想在切 topic 之前先看候选分支

不适合：

- 只是普通继续聊天时
- 明明已经知道要切到哪个 topic，却还先多做一次检查

### `memory_switch_topic`

作用：

- 把当前活跃上下文切换到一个已存在的 topic
- 更新会话里的 active topic 状态
- 让后续上下文装配围绕这个 topic 进行

典型触发时机：

适合什么时候用：

- 只有当你明确需要控制上下文时才用
- 普通使用时，很多切换应当是无感的
- 用户明确说“切到刚才那个话题”“回到旅游那条线”“我们单独聊这个分支”
- `memory_list_topics` 已经找到了目标 topic，下一步需要真正切过去

不适合：

- 只是想“查一下以前聊过没有”，那更像 `memory_search`
- 目标 topic 还不存在时，这时应该让对话自然进入新 topic，而不是强行切换一个旧 id

### `memory_save_fact`

作用：

- 把一个稳定事实写入 SQLite
- 这类事实会在后续合适的时候被召回
- 适合保存偏好、路径、约束、环境信息、项目决策等 durable memory

典型触发时机：

适合什么时候用：

- 用户说“记住这个”
- 某个路径、约束、偏好、账号引用以后还会反复用到
- agent 刚刚确认了一个后面大概率还要复用的稳定事实
- 希望某个事实即使 topic 漂移之后也还能被找回

不适合：

- 一次性的闲聊内容
- 明显只对当前一句回复有意义的中间态
- 不应该直接写入或暴露的敏感 secret 原文

### `memory_compact_topic`

作用：

- 重新整理并刷新某个 topic 的摘要
- 让这个分支的“短摘要/长摘要/open loops”更贴近当前状态
- 本质上是在维护 topic 级上下文质量

典型触发时机：

适合什么时候用：

- 某条分支最近变化很大
- 你希望它的摘要更贴近当前状态
- 一个话题已经聊了很多轮，方向明显变化了
- 你准备把某个 topic 交给后续 agent / 后续轮次继续使用

不适合：

- 每一轮都调用，没必要
- topic 还很短、状态也没明显变化时

### `memory_search`

作用：

- 在当前会话边界内搜索已存的 topics 和 durable facts
- 帮 agent 在提问用户之前，先检查答案是不是已经存在
- 是“先找再问”的核心工具

典型触发时机：

适合什么时候用：

- agent 怀疑某个事实之前已经记录过
- 不想先问用户“你再说一遍”
- 用户问“我之前说过什么来着”“那个偏好你还记得吗”
- agent 需要恢复旧事实，但不知道它具体属于哪个 topic

不适合：

- 已经明确知道目标 topic，并且只是要切回去继续聊
- 把它当成跨用户、跨 agent 检索工具使用，这不符合隔离边界

## 最常见的触发顺序

实际使用里，常见的是这几种组合：

1. 用户要回到旧话题：
   `memory_list_topics` -> `memory_switch_topic`
2. 用户要记住一个稳定事实：
   `memory_save_fact`
3. agent 怀疑以前记过，但不确定在哪：
   `memory_search`
4. 某个 topic 聊了很多轮、摘要过时：
   `memory_compact_topic`

更推荐的原则是：

- 先按语义判断，再选工具
- 不是每轮都要调工具
- 能无感恢复时，就不要把对话搞成工具操作流程

## 什么内容值得记

很值得记：

- 路径
- 环境信息
- 账号引用
- 安全约束
- 项目决策
- 长期偏好

不太值得记：

- 一次性的闲聊
- 短期中间态
- 不应该直接暴露的 secret 原文

## 对用户来说，正确的效果是什么

正确的用户感受一般是：

- “它确实记得我们刚才在聊什么。”
- “它没有逼我再重复一次偏好或信息。”
- “中途被打断以后，它还是能回到正确上下文。”
- “不同用户、不同 agent 之间不会串记忆。”

而不是：

- “它每次都告诉我它调用了什么工具。”
- “它开始像数据库一样回显一堆内部字段。”
- “别的用户能问出我刚保存的私密信息。”
