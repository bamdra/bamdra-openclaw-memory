# bamdra-memory 使用指南

## 别把它当成“数据库功能”，把它当成“连续性增强”

`bamdra-memory` 的价值不在于多了几个工具名，而在于对话会变得更像和一个持续工作的搭档协作。

最理想的体验是：

- 你中途被别的话题打断，也不用担心前面的思路丢掉
- 你说“记住这个”，后面真的能用上
- 你重新聊回之前的话题时，agent 能自然接上

## 一个典型使用过程

假设你正在和 OpenClaw 一起规划生活和工作：

### 场景一：隐式切换又自然接回

你说：

> 下个月去哪边旅游比较好？

接着又说：

> 如果去大阪，有哪些一定要吃的东西？

这时突然出现一个完全不同的话题：

> 我刚收到一个工作邮件，帮我写个礼貌回复，说我明天上午发文件过去。

等邮件处理完，你继续问：

> 继续说旅游。如果只有一个短周末，大阪和京都哪个更适合吃东西？

理想效果：

- 旅游主线还能自然接上
- “吃什么”这条支线仍然和旅游主线保持关联
- 工作邮件不会污染后面的旅游建议

### 场景二：显式要求记住一个事实

你说：

> 请记住我坐火车更喜欢靠窗的位置。

理想效果：

- 这个偏好会被保存下来
- 后面聊旅游、订票时不需要你再重复一次

## 实际上有哪些工具

### `memory_list_topics`

适合什么时候用：

- 想看当前 session 已经有哪些上下文线程
- 不确定之前那段对话现在被归到了哪里

### `memory_switch_topic`

适合什么时候用：

- 只有当你明确需要控制上下文时才用
- 普通使用时，很多切换应当是无感的

### `memory_save_fact`

适合什么时候用：

- 用户说“记住这个”
- 某个路径、约束、偏好、账号引用以后还会反复用到

### `memory_compact_topic`

适合什么时候用：

- 某条分支最近变化很大
- 你希望它的摘要更贴近当前状态

### `memory_search`

适合什么时候用：

- agent 怀疑某个事实之前已经记录过
- 不想先问用户“你再说一遍”

## 什么内容值得记

很值得记：

- 路径
- 环境信息
- 账号引用
- 安全约束
- 项目决策
- 长期偏好

不太值得记：

- 一次性的闲聊
- 短期中间态
- 不应该直接暴露的 secret 原文

## 对用户来说，正确的效果是什么

正确的用户感受一般是：

- “它确实记得我们刚才在聊什么。”
- “它没有逼我再重复一次偏好或信息。”
- “中途被打断以后，它还是能回到正确上下文。”

而不是：

- “它每次都告诉我它调用了什么工具。”
- “它开始像数据库一样回显一堆内部字段。”
