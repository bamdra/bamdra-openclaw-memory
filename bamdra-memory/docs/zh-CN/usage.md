# bamdra-memory 使用指南

## 别把它当成“数据库功能”，把它当成“连续性增强”

`bamdra-memory` 的价值不在于多了几个工具名，而在于对话会变得更像和一个持续工作的搭档协作。

最理想的体验是：

- 你切 topic 时不用担心上下文污染
- 你说“记住这个”，后面真的能用上
- 你说“回到刚才那条线”，agent 知道你在说什么

## 一个典型使用过程

假设你正在和 OpenClaw 一起做项目：

### 第一段：开始主题 A

你说：

> 先聊 OpenClaw memory 的 SQLite 设计。

此时系统会倾向于把这段对话归到一个 SQLite 相关的 topic。

### 第二段：临时切到主题 B

你又说：

> 现在切到 Redis 作为可选缓存。

这时通常会形成另一个 topic，而不是把 Redis 混进 SQLite 的上下文里。

### 第三段：保存一个长期有用的事实

你说：

> 记住主库路径是 `/Users/mac/.openclaw/memory/main.sqlite`。

理想效果：

- 这个路径会被当成稳定事实保存
- 之后不需要你反复重复

### 第四段：切回主题 A

你说：

> 回到刚才 SQLite 那条线。

理想效果：

- agent 会恢复到 SQLite 分支
- 当前上下文以 SQLite 为主
- Redis 只留在另一条分支里

## 实际上有哪些工具

### `memory_list_topics`

适合什么时候用：

- 想看当前 session 已经有哪些分支
- 不确定“刚才那条线”具体叫什么

### `memory_switch_topic`

适合什么时候用：

- 用户明确要求切回某个旧分支
- agent 已经知道要回哪个 topic

### `memory_save_fact`

适合什么时候用：

- 用户说“记住这个”
- 某个路径、约束、偏好、账号引用以后还会反复用到

### `memory_compact_topic`

适合什么时候用：

- 某条分支最近变化很大
- 你希望它的摘要更贴近当前状态

### `memory_search`

适合什么时候用：

- agent 怀疑某个事实之前已经记录过
- 不想先问用户“你再说一遍”

## 什么内容值得记

很值得记：

- 路径
- 环境信息
- 账号引用
- 安全约束
- 项目决策
- 长期偏好

不太值得记：

- 一次性的闲聊
- 短期中间态
- 不应该直接暴露的 secret 原文

## 对用户来说，正确的效果是什么

正确的用户感受一般是：

- “它确实记得我们刚才哪条线在做什么。”
- “它没有逼我再重复一次路径。”
- “它回到旧 topic 的时候没有串台。”

而不是：

- “它每次都告诉我它调用了什么工具。”
- “它开始像数据库一样回显一堆内部字段。”
