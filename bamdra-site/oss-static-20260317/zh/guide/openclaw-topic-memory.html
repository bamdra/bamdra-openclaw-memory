bamdra-openclaw-memory ​

产品概述 ​

bamdra-openclaw-memory 是 Bamdra OpenClaw 记忆套件中的主插件。它围绕真实协作中的核心问题来设计：话题切换、中断恢复、稳定事实保存、摘要更新，以及如何让上下文在生产环境中保持足够精简。

一句话理解，它就是 OpenClaw 的上下文调度层: 帮助系统判断当前应该延续什么、切换什么、召回什么，而不是让用户频繁学习一套显式的“记忆操作方式”。

它可以独立运行；同时通过 npm 安装时，也能自动补齐 bamdra-user-bind，这样用户不需要为了身份层再手工执行第二次安装。

它解决的核心问题 ​

在没有记忆系统的情况下，OpenClaw 很容易在长会话里丢失重点。用户需要反复重复背景，稳定事实无法持续复用，之前做出的判断也容易在中断之后被冲散。bamdra-openclaw-memory 解决的正是这个断层，让 OpenClaw 从单轮回答能力，升级为可持续协作能力。

核心能力 ​

规则 + 自然语言驱动的双线保障 ​

插件的核心上下文调度机制，是两条线一起工作：

  • 一条线用明确规则做兜底，处理显式切换、稳定事实、边界控制和关键触发条件
  • 一条线用自然语言语义做判断，让 topic 切换、上下文恢复和事实召回更贴近日常对话

这使得它既不会僵硬到只能识别固定命令，也不会松散到完全不可预测。

话题连续性 ​

把不同分支隔离开来，并在用户回到旧话题时恢复正确上下文。

持久化事实召回 ​

把偏好、路径、约束、决策和引用信息保存下来，供后续继续使用。

紧凑上下文装配 ​

围绕当前活跃分支生成更精炼的 prompt，而不是不断重放全部历史。

显式工具控制 ​

在需要的时候，operator 可以显式查看 topic、切换分支、保存事实、刷新摘要和搜索记忆。

它带来的产品价值 ​

  • 让长会话真正可持续
  • 降低重复背景说明的成本
  • 让环境信息和项目事实可以长期复用
  • 为 OpenClaw 从实验阶段走向生产使用提供更清晰的基础

架构特点 ​

  • 默认使用 SQLite 作为事实源
  • 默认使用内存缓存作为热路径
  • 插件层保持轻量，便于接入 OpenClaw
  • 通过摘要与 topic 状态控制上下文规模

适合哪些团队 ​

如果你的团队符合下面这些特征，这个产品会很适合：

  • 使用 OpenClaw 进行长周期协作
  • 会频繁回到之前讨论过的主题
  • 希望稳定事实只保存一次、长期复用
  • 既需要自动恢复，也希望保留人工控制能力
